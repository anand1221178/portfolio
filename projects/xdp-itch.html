Project Overview

A high-performance, kernel-bypass market data feed handler for the NASDAQ ITCH 5.0 protocol. By utilizing XDP (eXpress Data Path) and eBPF, this engine processes packets entirely within the NIC driver, bypassing the Linux kernel network stack and recv() syscall overhead to achieve sub-microsecond latency.

Packets arrive at the NIC, the XDP hook fires the eBPF program, which parses Ethernet/IPv4/UDP/MoldUDP64/ITCH headers, filters by symbol using a BPF hash map, and pushes matched messages to a BPF ring buffer. A userspace C++ order book maintains live best bid/ask. Unmatched symbols are dropped in-kernel without ever waking userspace.

Key Features

Full Protocol Stack

Parses 5 protocol layers (Ethernet, IPv4, UDP, MoldUDP64, ITCH 5.0) entirely within the eBPF program at the XDP hook point.

Symbol Filtering

BPF hash map-based symbol filtering drops irrelevant stocks in-kernel. Only matched messages reach userspace via zero-copy ring buffer.

BPF Verifier Safe

All pointer arithmetic includes bounds checks to pass the BPF verifier. Handles variable-length MoldUDP64 message streams with bounded loop iteration.

Live Order Book

Price-level aggregated C++ order book with O(1) active order lookup. Handles Add, Execute, and Delete messages maintaining real-time best bid/ask.

Technical Implementation

1. XDP Packet Processing Pipeline

The eBPF program uses a cursor-based parser that advances through each protocol layer with verifier-safe bounds checking at every step. Non-IPv4, non-UDP, and wrong-port packets are passed to the kernel stack. Market data packets are parsed and filtered in-kernel.

SEC("xdp")
int xdp_itch_parser(struct xdp_md *ctx) {
    void *data_end = (void *)(long)ctx->data_end;
    void *data = (void *)(long)ctx->data;
    struct hdr_cursor nh = { .pos = data };

    // Parse Ethernet -> IPv4 -> UDP -> MoldUDP64
    if (parse_ethhdr(&nh, data_end, &eth) < 0) return XDP_PASS;
    if (eth_type != bpf_htons(ETH_P_IP)) return XDP_PASS;
    if (parse_ip4hdr(&nh, data_end, &iph) < 0) return XDP_PASS;
    if (ip_protocol != IPPROTO_UDP) return XDP_PASS;

    return XDP_DROP; // Consumed by eBPF

2. BPF Ring Buffer Delivery

Matched ITCH messages are pushed to userspace via a BPF ring buffer — a lock-free, zero-copy mechanism. The userspace reader polls the ring buffer and feeds messages directly into the order book engine.

// Kernel side: reserve, copy, submit
void *ring_slot = bpf_ringbuf_reserve(&itch_ringbuf,
    sizeof(struct itch_add_order), 0);
if (ring_slot) {
    __builtin_memcpy(ring_slot, itch_msg,
        sizeof(struct itch_add_order));
    bpf_ringbuf_submit(ring_slot, 0);
}

3. Order Book Engine

A C++ price-level aggregated order book using std::map for price levels (sorted by price) and std::unordered_map for O(1) order lookup. Handles Add ('A'), Execute ('E'), and Delete ('D') messages, maintaining live best bid/ask spread.

void OrderBook::add_order(uint64_t order_ref, char side,
                          uint32_t price, uint32_t shares) {
    auto& level = (side == 'B') ? bids[price] : asks[price];
    level.total_volume += shares;
    level.order_queue.push_back({order_ref, shares});
    // O(1) lookup stored in active_orders hash map
}

Results

Sub-Microsecond Latency

Achieved 930 ns average per-packet eBPF processing latency, measured by the kernel via bpftool prog show. This is the actual CPU time spent in the XDP program per packet invocation.

Zero-Copy Architecture

By processing packets at the NIC driver level and using BPF ring buffers, the entire path from wire to order book avoids kernel-to-userspace copies. Symbol filtering in-kernel means irrelevant packets never consume userspace CPU cycles.

Conclusion

This project demonstrates that eBPF/XDP is a viable software fast path for latency-sensitive financial infrastructure. The sub-microsecond parsing latency, combined with in-kernel symbol filtering and zero-copy delivery, provides a practical alternative to FPGA-based feed handlers at a fraction of the development complexity.
