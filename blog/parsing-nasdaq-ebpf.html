Step Zero: Read the Spec Before Writing a Line of Code

Before I touched any code, I spent time reading two specifications cover to cover: the MoldUDP64 transport protocol spec (~5 pages) and the NASDAQ ITCH 5.0 message format spec (Section 4, message formats). This sounds obvious, but it's the step most people skip. I almost skipped it too.

MoldUDP64 is the wrapper NASDAQ uses to multiplex ITCH messages into UDP packets. The header is 20 bytes: 10-byte session ID, 8-byte sequence number, 2-byte message count. After that, each ITCH message is prefixed with a 2-byte length field. A single UDP packet can carry multiple ITCH messages. This detail matters enormously for the XDP parser — more on that later.

The ITCH Add Order message ('A') is 36 bytes. The key insight from the spec: Stock Locate (offset +1, 2 bytes) is a daily integer index for each symbol. It's meant to be used as an array key, not the 8-byte ASCII stock name. This became my BPF hash map key — a u16 lookup instead of an 8-byte string compare.

Week One: Learning XDP the Hard Way

I knew nothing about eBPF when I started. I worked through the xdp-tutorial exercises: packet01 (parsing Ethernet/IP/UDP headers), packet02 (bounds checking patterns), and packet03 (XDP_PASS/XDP_DROP logic). This took a full week.

Early on, I thought the BPF verifier did "static analysis at compile time." Wrong. The verifier runs at load time — when you load the eBPF bytecode into the kernel, not when clang compiles it. So you can have code that compiles perfectly but gets rejected by the kernel. Two separate gatekeepers: 

clang compiles your C    -->  produces eBPF bytecode (.o file)
                                       |
you load it into kernel  -->  VERIFIER RUNS HERE (once, at load time)
                                       |
packet arrives           -->  already-verified program executes, full speed

This is why eBPF is both fast and safe. The verifier does all the hard work once upfront — proves your program can't crash the kernel, can't loop forever, always bounds-checks memory — and then gets out of the way forever. At packet time, there's no VM interpretation either: the JIT compiles eBPF bytecode to native x86 instructions.

The Parsing Chain I Almost Got Wrong

After the tutorials, I tried to explain the full packet journey and wrote:

"Ethernet → IP → UDP → ITCH message"

Missing a layer. Between UDP and ITCH sits MoldUDP64. After the UDP header, you're not staring at an ITCH message — you're staring at a MoldUDP64 packet containing potentially multiple ITCH messages, each length-prefixed. Skip this layer and you parse garbage.

The corrected memory map from the UDP payload: 

UDP Payload:
[+0  - +9 ]  MoldUDP64 Session        (10 bytes, skip)
[+10 - +17]  MoldUDP64 Sequence Num   (8 bytes,  skip)
[+18 - +19]  MoldUDP64 Message Count  (2 bytes,  READ)
[+20 - +21]  Message Block Length     (2 bytes,  READ)
[+22]        ITCH Message Type        (1 byte,   BRANCH POINT)

At byte +22 you read one character — 'A', 'D', 'E' — and that tells you which ITCH message you're holding and how to parse the rest.

The Verifier's Loop Problem

MoldUDP64's Message Count says there are N messages in this packet. My first instinct: while(messages_remaining--). The verifier rejected it instantly.

The verifier can't prove termination of a loop bounded by packet data — an attacker could put any value in Message Count. The solution: a fixed compile-time bound with an early break: 

#define MAX_ITCH_MESSAGES 64

for (int i = 0; i < MAX_ITCH_MESSAGES; i++) {
    if (i >= msg_count) break;  // runtime exit
    // parse message...
}

The verifier sees 64 iterations maximum — provably finite. The break handles the actual count at runtime. This pattern shows up everywhere in eBPF: fixed bounds to satisfy the verifier, runtime checks for actual logic.

In-Kernel Symbol Filtering

NASDAQ sends data for every listed stock. A trading firm only cares about a few hundred. In a traditional architecture, every packet reaches userspace and gets filtered there — wasting cycles on irrelevant data.

In XDP, I populate a BPF hash map from userspace with the stock_locate codes I care about. When an ITCH message is parsed, I look up the Stock Locate. If it's not tracked, the message never leaves the kernel: 

__u16 stock_loc = bpf_ntohs(itch_msg->stock_locate);
__u8 *track = bpf_map_lookup_elem(&stock_filter, &stock_loc);
if (track && *track == 1) {
    void *slot = bpf_ringbuf_reserve(&itch_ringbuf,
        sizeof(struct itch_add_order), 0);
    if (slot) {
        __builtin_memcpy(slot, itch_msg,
            sizeof(struct itch_add_order));
        bpf_ringbuf_submit(slot, 0);
    }
}
// No match? Silently dropped. Userspace never knows.

The bpf_ntohs() is critical — all ITCH integers are big-endian (network byte order) and x86 is little-endian. Miss one byte-swap and your prices are garbage. I learned this the hard way.

The Benchmark Script That Broke Twice

The first version of the benchmark script had two bugs that perfectly illustrate the kind of system-level thinking this project demands.

Bug 1: Reading stats from a dead program. The script sent SIGINT to the eBPF reader (which detaches from the NIC and unloads the program), then asked bpftool for the stats. But the program was already destroyed. Fix: grab stats before killing the reader.

Bug 2: Empty simulator logs. C's printf is fully buffered when piped to a file. The timeout command kills the simulator before the buffer flushes, so the log stays empty. Fix: stdbuf -oL forces line-buffered output.

Both bugs have the same root cause: you're orchestrating multiple processes (reader, simulator, bpftool) with shared kernel state, and order of operations matters at the system level. This is the kind of debugging you don't encounter in application programming.

The Result

The final benchmark across 500K+ packets: 

======================================================
               eBPF LATENCY RESULTS
======================================================
Total Packets Processed: 500469
Total eBPF CPU Time:     465851581 ns
------------------------------------------------------
Average Latency:         930 nanoseconds/packet
======================================================

Sub-microsecond. That's parsing 5 protocol layers, filtering via hash map, and delivering to userspace via ring buffer — all within the NIC driver, before the kernel network stack sees the packet.

A traditional recv() path costs 2,000-5,000 nanoseconds. We eliminated 60-80% of the operating system's overhead. To go faster than this in software, you'd need... well, you'd need hardware. FPGAs hit 50-100ns, but at orders of magnitude more development cost.

What I Actually Learned

  • Read the spec first. I almost started coding without understanding MoldUDP64. That would have cost me days of debugging garbled messages. Twenty minutes reading a 5-page spec saved me hours.
  • The verifier is your teacher, not your enemy. Every rejection forced me to think about pointer safety at a level I'd never considered. Can this pointer go out of bounds? Can this loop run forever? The final code is provably safe — not just tested, proven.
  • The tutorial was non-negotiable. A week doing the XDP tutorial exercises felt slow. But when I sat down to write the actual parser, the bounds-checking pattern was automatic. I didn't fight the verifier on my main code because I'd already fought it 50 times in practice.
  • Endianness will haunt you. Network protocols are big-endian. x86 is little-endian. bpf_ntohs(), bpf_ntohl(), be64toh() — you will call these more times than you thought possible.
  • System-level debugging is different. The benchmark bugs weren't logic errors — they were orchestration errors. Process lifecycle, buffer flushing, kernel state — you're debugging the system, not the code.

Is This Production-Ready?

No. A production feed handler needs per-packet tail latency profiling, gap detection for dropped sequences, TCP retransmission for SOUP-TCP feeds, and the order book should use flat arrays indexed by price tick instead of std::map. But the architecture is sound — the same pattern Cloudflare uses for DDoS mitigation and Meta uses for load balancing, applied to market data.
