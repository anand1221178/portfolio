The Latency Tax of recv()

Here's how a traditional market data feed handler works: a UDP packet arrives at the NIC, travels up through the kernel network stack (sk_buff allocation, protocol processing, socket buffer), your application calls recv() (syscall overhead, context switch), copies the data to userspace, then finally parses the ITCH protocol and updates the order book.

Every step adds latency. The recv() syscall alone is 1-2 microseconds. Buffer copies add more. For high-frequency trading, where you're racing other firms to react to market data, every microsecond matters.

The question I wanted to answer: what if you could parse the market data before it reaches userspace?

XDP: Processing Packets at the NIC Driver

XDP (eXpress Data Path) is a hook in the Linux kernel that runs an eBPF program at the earliest possible point in the network stack — right in the NIC driver, before the kernel even allocates an sk_buff. Your program gets a raw pointer to the packet data and can inspect, modify, redirect, or drop the packet.

The catch: eBPF programs must be provably safe. The BPF verifier statically analyzes every possible execution path to ensure you never dereference an out-of-bounds pointer, never loop forever, and never access kernel memory you shouldn't. This is not a suggestion — if the verifier rejects your program, it doesn't load.

Parsing 5 Protocol Layers in eBPF

NASDAQ market data arrives as: Ethernet → IPv4 → UDP → MoldUDP64 → ITCH 5.0. Each layer has a variable-length header that must be parsed and bounds-checked before accessing the next.

I used a cursor-based parser: a struct hdr_cursor that tracks the current position in the packet. Each parse function advances the cursor past its header and returns the next protocol type. At every step, a bounds check satisfies the verifier. 

// Each layer: bounds check, parse, advance cursor
static __always_inline int parse_ethhdr(
    struct hdr_cursor *nh, void *data_end,
    struct ethhdr **ethhdr)
{
    struct ethhdr *eth = nh->pos;
    if (eth + 1 > data_end)      // verifier needs this
        return -1;
    nh->pos = eth + 1;           // advance past ethernet
    *ethhdr = eth;
    return eth->h_proto;         // tells us: IPv4? IPv6?
}

The tricky part is MoldUDP64. It's NASDAQ's transport protocol that multiplexes multiple ITCH messages into a single UDP packet. Each message is prefixed with a 2-byte length field. I iterate through messages with a bounded loop (MAX_ITCH_MESSAGES=64) because the BPF verifier doesn't allow unbounded loops.

In-Kernel Symbol Filtering

NASDAQ sends data for every listed stock. A trading firm typically only cares about a few hundred symbols. In a traditional architecture, every packet reaches userspace and gets filtered there — wasting CPU cycles on irrelevant data.

In XDP, I use a BPF hash map populated from userspace with the stock_locate codes of symbols we care about. When an ITCH Add Order message is parsed, I check the hash map. If the symbol isn't tracked, the message is silently dropped — it never leaves the kernel, never wakes userspace, never consumes a ring buffer slot. 

__u16 stock_loc = bpf_ntohs(itch_msg->stock_locate);
__u8 *track = bpf_map_lookup_elem(&stock_filter, &stock_loc);
if (track && *track == 1) {
    // This stock matters - push to ring buffer
    void *slot = bpf_ringbuf_reserve(&itch_ringbuf,
        sizeof(struct itch_add_order), 0);
    if (slot) {
        __builtin_memcpy(slot, itch_msg,
            sizeof(struct itch_add_order));
        bpf_ringbuf_submit(slot, 0);
    }
}
// Else: silently dropped. Userspace never knows.

The Order Book

On the userspace side, a C++ order book consumes messages from the BPF ring buffer. It's a price-level aggregated book using std::map (sorted by price) with an std::unordered_map for O(1) order lookup by reference number. Handles Add, Execute, and Delete messages and maintains a live best bid/ask spread.

The C++ book is wrapped in extern "C" functions so the C-based eBPF reader can call it without a C++ runtime dependency in the loader. A small detail, but the kind of thing that matters when you're mixing kernel C with userspace C++.

The Result: 930 Nanoseconds

I measured latency using bpftool prog show, which reports the kernel's own measurement of total CPU time spent in the eBPF program divided by invocation count. Across 500K+ packets:

Average latency: 930 nanoseconds per packet

That's sub-microsecond parsing of 5 protocol layers, symbol filtering via hash map lookup, and zero-copy delivery to userspace via ring buffer. All within the NIC driver, before the kernel network stack even sees the packet.

What I Learned

  • The BPF verifier is your enemy and your friend. It rejected my program dozens of times. Every rejection forced me to think more carefully about pointer safety. The final code is provably safe — no buffer overflows, no OOB reads, guaranteed.
  • __always_inline is not optional. eBPF programs have a maximum instruction count. Without inlining, function calls blow the budget. Every helper function must be __always_inline.
  • Big-endian is everywhere. Network protocols are big-endian. x86 is little-endian. You will call bpf_ntohs() and bpf_ntohl() more times than you thought possible. Miss one and your prices are garbage.
  • Ring buffers are the right abstraction. BPF ring buffer (BPF_MAP_TYPE_RINGBUF) is lock-free, supports variable-size entries, and gives zero-copy access to userspace. It replaced the older perf buffer for good reason.
  • eBPF is the future of networking. What I built in a few weeks would have required a kernel module or FPGA five years ago. The eBPF ecosystem (libbpf, CO-RE, bpftool) has matured to the point where kernel-level packet processing is accessible to application developers.

Is This Production-Ready?

No. A production feed handler needs per-packet tail latency profiling (not just averages), gap detection for dropped sequences, TCP retransmission handling for SOUP-TCP feeds, and integration with a risk system. The order book uses std::map which allocates — a real HFT book uses flat arrays indexed by price tick.

But the architecture is sound. eBPF/XDP for parsing and filtering, ring buffer for delivery, C++ for the book. This is the same pattern that Cloudflare uses for DDoS mitigation and Meta uses for load balancing. Applied to market data, it's a viable software alternative to FPGA-based feed handlers — at a fraction of the development cost and with far better debuggability.
