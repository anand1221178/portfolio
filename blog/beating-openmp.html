How I Beat OpenMP with a Custom Work-Stealing Scheduler

May 2026 HPC, NUMA, Systems 15 min read

Project

TopoSteal Project Page Source Code

Key Numbers

1.17x Speedup
5% Over OpenMP
85% Local Steals

The Starting Point: A Bare-Metal Work-Stealing Deque

I'd already built a lock-free work-stealing thread pool for a bare-metal Raspberry Pi 4 using the Chase-Lev deque algorithm. It worked. But it treated all cores as equidistant when picking steal victims — random selection, no topology awareness.

The idea for TopoSteal came from a simple question: what if the scheduler knew which cores were close and which were far? On a dual-socket Xeon, accessing local DRAM costs ~80ns. Crossing to the remote socket costs ~140ns. That's a 1.75x penalty. Uniform random stealing ignores this and picks a remote victim 50% of the time.

Phase 1: Building on macOS, Targeting Linux

I started development on my Mac. macOS can't do perf_event_open or pthread_setaffinity_np, so I built a mock topology system — fake distance matrices that simulated a 2-socket NUMA system. The mock let me develop and stress-test the core logic (deques, weighted victim selection, CDF sampling) without access to real hardware.

Then my Linux PC fried itself. So I stayed on macOS longer than planned, building the full library with mock topology while waiting for access to the CHPC Lengau cluster in South Africa — a dual-socket Xeon E5-2690v3 with real NUMA.

The hwloc Discovery Layer

On real hardware, TopoSteal uses hwloc to walk the CPU tree and build a pairwise distance matrix. For each pair of cores, it finds their lowest common ancestor in the topology tree:

  • Common ancestor is an L2 cache → distance 1 (very close)
  • Common ancestor is an L3 cache → distance 4 (same socket)
  • Common ancestor is the machine root → distance 10 (cross-NUMA)

The hwloc integration had a critical design decision: the hwloc_topology_t handle is a static inside topo.c, never exposed in the header. If it lived in topo.h, every file that includes the header would pull in hwloc as a dependency even though they never use it. Encapsulation matters.

The 1/d² Weighting

Given the distance matrix, steal weights are computed using inverse-square weighting: wij = 1/dij². On our test system (distances 4 for same-socket, 10 for cross-socket), this gives an 86% probability of selecting a same-socket victim. I tried 1/d first — too weak. 1/d² is aggressive enough to matter. 

// Victim selection: single pass over the CDF
int weights_pick_victim(weights_t *w, int id, unsigned int *seed) {
    float r = (float)rand_r(seed) / RAND_MAX;
    for (int j = 0; j < w->num_workers; j++) {
        if (j == id) continue;
        if (r < w->steal_thresholds[id][j])
            return j;
    }
    return -1;
}

First Cluster Run: The cpu_map Bug

I got access to Lengau, built the code on a compute node (which involved sed-fixing the Makefile every git pull because the cluster's hwloc was ancient and I'd built 2.9.3 from source in $HOME/local), and ran the first benchmark.

Result: zero speedup. Uniform and topology-weighted performed identically. I stared at distance matrices for two days.

The bug: hwloc logical core indices are not OS CPU IDs. On Lengau, the interleaving is: L#0→CPU0, L#1→CPU2, L#2→CPU4. I was pinning workers with CPU_SET(id, ...) but building the distance matrix using hwloc's logical indices. Workers thought they were on one socket but were physically on another. 

// The fix: 3 lines that changed everything
for (int i = 0; i < t->num_cores; i++) {
    hwloc_obj_t core = hwloc_get_obj_by_depth(topo, depth, i);
    hwloc_obj_t pu = hwloc_get_obj_inside_cpuset_by_type(
        topo, core->cpuset, HWLOC_OBJ_PU, 0);
    t->cpu_map[i] = (pu) ? (int)pu->os_index : i;
}

Three lines. cpu_map[i] translates logical index i to physical CPU Pn. Workers now pin correctly. Instant 1.15x speedup.

The Wake-Up Call: "Wasn't PMU the Entire Point?"

After the first round of pointer-chase benchmarks (1.15-1.18x speedup, 85% local steals), I was writing the paper's future work section: "Future work includes integration of the PMU feedback loop into the benchmark."

Wait. The PMU — the dynamic hardware counter feedback that adjusts steal probabilities at runtime based on cache miss rates — was supposed to be the paper's main contribution. I'd implemented it, but never actually benchmarked it. The benchmarks all used static topology weights.

I was about to publish a paper about a PMU-driven scheduler where the PMU was never turned on.

So I went back and wired it in properly. Three-mode evaluation: uniform random, static topology, and topology + PMU. The PMU sampler reads cache miss counters every 10ms via perf_event_open and rebuilds the steal probability table every 50ms using adjusted distances: d'ij = dij × (1 + miss_ratej).

On the pointer-chase benchmark, PMU feedback matched or slightly beat static weights — not a huge delta, because all workers have similar miss profiles. But the pipeline works end-to-end and on heterogeneous workloads the dynamic adaptation would provide a larger benefit.

SpMV: Beating OpenMP on a Real Workload

The pointer chase proved the mechanism. But it's a microbenchmark. I needed a real workload that Intel, AMD, and NVIDIA engineers would recognize immediately.

SpMV (Sparse Matrix-Vector multiply) is the kernel that every HPC team speaks fluently. I downloaded audikw_1 from SuiteSparse — 943K rows, 77 million nonzeros, from an Audi car body structural mechanics simulation.

The first SpMV attempt used thermal2 (only 7 nnz/row) — too sparse, all modes performed identically because the working set fit in cache after the first iteration. With audikw_1 at 82 nnz/row, each row does 82 random accesses into the x-vector, making NUMA placement critical.

The OpenMP comparison also needed fixing. My first version created/destroyed 24 threads per repetition while OpenMP reused a thread pool. Unfair. I switched to persistent workers with pthread_barrier_t to match OpenMP's thread reuse.

Final result: TopoSteal 7.46 GFLOP/s, OpenMP 7.09 GFLOP/s with schedule(static) + OMP_PROC_BIND=close. 5% faster.

5% doesn't sound like much until you realize that's Intel's own recommended configuration for SpMV. We beat it because our task placement creates imbalance (all tasks start on 2 of 24 workers), requiring dynamic rebalancing that static partitioning can't provide — and the topology-aware stealing keeps 82% of that rebalancing on the same NUMA node.

Graph500 BFS: The Benchmark That Failed

I also tried Graph500 BFS with R-MAT Kronecker graphs. The idea: power-law degree distribution creates unpredictable load imbalance, perfect for work-stealing.

Result: OpenMP schedule(dynamic) was 1.73x faster.

Why? BFS is level-synchronous — you need a barrier between every level. Each level requires deque initialization, task construction, push, barrier sync, and serial edge counting. The overhead of our task infrastructure per level overwhelmed any NUMA benefit. And OpenMP's schedule(dynamic) IS work-stealing — we were bringing a hand-rolled scheduler to a fight against GCC's production runtime.

I dropped BFS from the paper. Report what works AND what doesn't.

The Cluster Experience

Working on CHPC Lengau taught me things no local development can. Every new compute node resets your GCC module (module load gcc/7.3.0, every time). GCC 6.1's _Atomic doesn't work with -fopenmp. The hwloc Makefile paths get overwritten by every git pull. perf_event_paranoid=1 lets you read PMU counters without sudo. These aren't in any textbook — you learn them by watching things break at 2am on a PBS queue.

What I Actually Learned

  • Logical != physical. hwloc core indices are NOT OS CPU IDs. Build a cpu_map or your topology-aware scheduling is scheduling-unaware. This bug gave me zero speedup for two days.
  • 1/d vs 1/d² matters. Linear inverse distance was too weak — only 65% local steals. Inverse-square jumped to 85%. The weighting function is the tuning knob for the entire system.
  • Fair comparisons are hard. Thread creation/destruction overhead, barrier costs, first-touch memory placement — any of these can make your scheduler look better or worse than it actually is. I rewrote the SpMV benchmark three times before the comparison was honest.
  • Not every workload benefits. BFS was 1.73x slower. SpMV with sparse matrices (thermal2, 7 nnz/row) showed no difference. You need enough work per task and enough random memory access for NUMA to matter.
  • 5% over a production runtime is real. In the memory-bandwidth world, single-digit improvements are the norm. The same techniques on 4+ socket systems with deeper NUMA hierarchies would show proportionally larger gains.
  • Benchmark honestly. I almost shipped a paper about PMU feedback where the PMU was never turned on. If your "main contribution" isn't in your benchmark, you don't have a contribution.

What's Next

I'm working with Bryan Johnston (CHPC / Georgia Tech) to extend TopoSteal to multi-socket systems and GPU-heterogeneous topologies, targeting an SC (Supercomputing Conference) submission. The hypothesis: on 4+ socket systems where the NUMA tree is deeper and the PMU sees more variation in cache pressure across nodes, both the static weighting and dynamic feedback should show proportionally larger gains.

The code is on GitHub. The IEEE-format paper is in the repo. If you have access to interesting multi-socket hardware and want to help test it, reach out.
