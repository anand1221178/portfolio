The Problem with Uniform Stealing

Every major parallel runtime — Intel TBB, Cilk, Go's goroutine scheduler — uses work-stealing to balance load across cores. When a worker runs out of tasks, it steals from another worker's queue. The textbook approach: pick a victim uniformly at random.

On a single-socket machine, this is fine. Every core is roughly equidistant from every other core. But on a dual-socket Xeon with NUMA (Non-Uniform Memory Access), "equidistant" is a lie. Accessing memory on the local socket costs ~80ns. Crossing to the remote socket costs ~140ns. That's a 1.75x penalty on every cache miss.

Uniform random stealing doesn't know about this. It picks a remote victim 50% of the time. Half your steals pay the NUMA tax for no reason.

The Idea: Weight Victims by Topology Distance

The fix is conceptually simple: instead of picking victims uniformly, bias toward nearby cores. Use hwloc to discover the actual CPU topology at runtime — which cores share an L2 cache, which share an L3, which are across the NUMA boundary — and build a distance matrix. Then convert distances to steal probabilities.

I chose inverse-square weighting: wij = 1/dij². On our test system (distances 4 for same-socket, 10 for cross-socket), this gives an 86% probability of picking a same-socket victim. The math works out cleanly and the preference is aggressive enough to matter without starving remote cores entirely.

The Bug That Almost Killed the Project

First benchmark run: zero speedup. Uniform and topology-weighted performed identically. I spent two days staring at distance matrices before finding it.

hwloc logical core indices are not OS CPU IDs. On our Lengau cluster, hwloc reports core L#1 but the OS knows it as CPU 2. The interleaving is: L#0→CPU0, L#1→CPU2, L#2→CPU4, etc. I was pinning workers to CPU N using CPU_SET(id, ...) but building the distance matrix using hwloc's logical indices. Workers thought they were on one socket but were actually on another.

The fix: build a cpu_map[] that translates logical index to physical CPU by querying each core's processing unit OS index. Three lines of code. Instant 1.15x speedup. 

// The fix: query hwloc for the REAL cpu id
for (int i = 0; i < t->num_cores; i++) {
    hwloc_obj_t core = hwloc_get_obj_by_depth(hwloc_topo, depth, i);
    hwloc_obj_t pu = hwloc_get_obj_inside_cpuset_by_type(
        hwloc_topo, core->cpuset, HWLOC_OBJ_PU, 0);
    t->cpu_map[i] = (pu) ? (int)pu->os_index : i;
}

Adding PMU Feedback

Static topology weights are good. But what if cache pressure changes at runtime? A worker hammering DRAM is a bad steal target regardless of topology.

I added a background thread that reads hardware cache miss counters via perf_event_open every 10ms and adjusts the effective distance: d'ij = dij × (1 + miss_ratej). Workers under cache pressure become less attractive. The probability table gets rebuilt every 50ms.

On our homogeneous pointer-chase benchmark, PMU feedback provides only marginal additional gain (all workers have similar miss rates). But it proves the full pipeline works end-to-end, and on heterogeneous workloads the dynamic adaptation would provide a larger benefit.

Beating OpenMP on Real SpMV

The pointer-chase microbenchmark proved the mechanism. But I needed a real workload. SpMV (Sparse Matrix-Vector multiply) is the canonical memory-bound HPC kernel — it's in every Intel MKL benchmark suite.

I grabbed audikw_1 from SuiteSparse — 943K rows, 77 million nonzeros, from an Audi car body structural simulation. Loaded it in Matrix Market format, converted to CSR, and ran it through both TopoSteal and OpenMP with schedule(static) and OMP_PROC_BIND=close (Intel's recommended config).

Result: TopoSteal 7.46 GFLOP/s, OpenMP 7.09 GFLOP/s. 5% faster.

5% doesn't sound like much until you realize we're beating a production-quality runtime that's been optimized for decades. The reason it works: our task placement creates imbalance (all tasks start on 2 of 24 workers), requiring dynamic rebalancing. OpenMP's static scheduling can't adapt. TopoSteal's work-stealing rebalances the load while keeping 82% of steals on the same NUMA node.

What I Learned

  • Topology matters. On NUMA systems, the difference between a local and remote memory access is real and measurable. Ignoring it in your scheduler leaves performance on the table.
  • hwloc lies (sort of). Logical indices are not physical CPU IDs. Always build a cpu_map. This bug cost me two days.
  • Lock-free is hard but worth it. The Chase-Lev deque with C11 atomics took three attempts to get right. The sequential consistency fence between the bottom store and top load in pop is the kind of subtlety that doesn't show up in unit tests — it only fails under high contention on real hardware.
  • Benchmark honestly. I tried Graph500 BFS and it was 1.73x slower than OpenMP. The per-level barrier overhead dominated. I dropped it from the paper. Report what works AND what doesn't.
  • 5% over OpenMP is real. In the memory-bandwidth world, single-digit improvements are the norm. The same techniques on a 4+ socket system with deeper NUMA hierarchies would show proportionally larger gains.
